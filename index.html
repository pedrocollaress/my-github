Git and GitHub Concepts

Git is a distributed version control system designed to efficiently manage projects, allowing multiple developers to work simultaneously without overwriting each other's work. It records the history of changes, making it possible to revert modifications and compare versions.

GitHub is a cloud-based platform that hosts Git repositories, facilitating collaboration, code review, issue tracking, and continuous integration. It enables teams to work together in an organized way, using features like Pull Requests, Issues, and Actions.

Branch and Pull Request Flow

The workflow with branches is fundamental for collaborative development. It allows developers to work on new features, bug fixes, or experiments in isolation, without impacting the main codebase. This approach increases productivity, reduces conflicts, and keeps the main branch stable.

Below is a typical step-by-step guide for working with branches and Pull Requests (PRs) in Git and GitHub, including the main commands used at each stage:

  1. Update your local repository:
    git pull origin main
    Fetches and integrates the latest changes from the remote main branch.
  2. Create a new branch:
    git checkout -b feature/your-feature-name
    Creates and switches to a new branch for your work.
  3. Develop your feature or fix:
    Make your code changes, add new files, or modify existing ones as needed.
  4. Stage and commit your changes:
    git add .
    git commit -m "Describe your changes here"
    Stages and commits your modifications with a descriptive message.
  5. Push your branch to GitHub:
    git push origin feature/your-feature-name
    Uploads your branch and commits to the remote repository.
  6. Open a Pull Request (PR):
    Go to GitHub and open a PR from your branch to main. Describe your changes and request a review from your team.
  7. Review and address feedback:
    Team members review your PR. You may need to make additional commits to address comments.
  8. Merge the PR:
    Once approved, the PR is merged into main (either via GitHub or using git merge locally).
  9. Delete the feature branch (optional):
    git branch -d feature/your-feature-name
    git push origin --delete feature/your-feature-name
    Cleans up your local and remote branches after merging.

This step-by-step flow ensures that development is organized, traceable, and that the main branch remains production-ready. Using branches and PRs also facilitates code review, continuous integration, and collaborative problem-solving.

Conflict Resolution

Merge conflicts are a common part of collaborative development with Git. They occur when two branches have made changes to the same lines in a file, or when a file has been deleted in one branch and modified in another. Git cannot automatically decide which change to keep, so manual intervention is required.

Here is a step-by-step guide to identifying and resolving merge conflicts, including the main commands used:

  1. Attempt to merge branches:
    git checkout main
    git pull origin main
    git merge feature/your-feature-name
    Try to merge your feature branch into main. If there are conflicts, Git will notify you.
  2. Identify conflicting files:
    git status
    Lists the files with conflicts. Open these files to see conflict markers (<<<<<<<, =======, >>>>>>>).
  3. Manually resolve conflicts:
    Edit the conflicting files, choosing which changes to keep or combining them as needed. Remove the conflict markers after resolving.
  4. Mark conflicts as resolved:
    git add <file>
    Stage the resolved files.
  5. Complete the merge:
    git commit
    Finalize the merge. Git may auto-generate a commit message for the merge resolution.
  6. Push the resolved changes:
    git push origin main
    Update the remote repository with the resolved merge.

Tips:

  • Use git log --merge to see commits causing conflicts.
  • Graphical tools like VS Code, GitKraken, or git mergetool can help visualize and resolve conflicts.
  • Always test your code after resolving conflicts to ensure nothing is broken.

Careful conflict resolution is essential to maintain code integrity and team productivity.
